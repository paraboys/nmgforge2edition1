00 · The mission & the method

Don't write the app. Orchestrate the agents that write it.

In the qualifier you proved you could set up Hermes + OpenClaw and ship a tiny build. Today is the real thing: you run a proper agile sprint cycle with AI agents and ship a non-trivial, multi-tenant SaaS in a day.

The method we want you to practise is exactly how modern AI-native teams ship — “agile, with bots.” Here's the loop in one breath:

spec  →  tight sprint (a few scoped issues)  →  Hermes plans & assigns in Slack  →  OpenClaw codes + tests + reports  →  CI runs  →  you (human) review & merge the PR  →  next sprint.

The principles we're grading you on (straight from how a real AI-native shop runs this):

  • Keep sprints tight and focused. Small, well-scoped sprints beat one giant “build everything” prompt every time. A bloated sprint produces slop.
  • Two roles, in the open. Hermes is the product owner / orchestrator. OpenClaw is the coder. They never talk privately — every handoff happens in Slack so a human (you) stays in the loop.
  • Right model for the right job. Use a stronger reasoning model for planning/architecture and a fast coding model for implementation (see EastRouter §02). Cluster your tasks by difficulty.
  • Humans merge to main. Agents open PRs; you review and merge release candidates. No bot auto-merges to main — bots are great at scope creep, so you gate it.
  • Stay consistent. Once an agent config works, don't keep “upgrading” it mid-event — it creates chaos. Lock it, branch a new config if you must experiment.
What we are really testing: not “can you write a SaaS” — but can you make two agents collaborate, under your direction, to produce working software with a clean process and a clear audit trail. A smaller app with a flawless, well-evidenced agent loop beats a bigger app with no visible orchestration.

03 · The build

PulseDesk — a multi-tenant support-desk SaaS.

Build PulseDesk: a hosted help-desk / support-ticketing product that multiple companies (tenants) use to manage customer support. Think a focused mini-Zendesk. It's deliberately chosen because it has real depth (multi-tenancy, roles, SLAs, metrics) and decomposes cleanly into tight sprints.

Features are tiered. Must = the core you should land. Should = depth that lifts you. Stretch = wow. You are not expected to finish everything — get the Must tier solid, then go as deep as your sprints allow.

Must Core — land these first

  • Multi-tenancy. Organizations (tenants). Every record belongs to an org; a user from Org A can never see Org B's data. (Scope every query by organization_id.)
  • Auth & roles (Laravel Sanctum). Register / login. Roles: admin, agent, customer.
  • Tickets CRUD. Fields: subject, description, status (open / pending / resolved / closed), priority (low / medium / high / urgent), requester (customer), assignee (agent), tags/labels, created/updated timestamps.
  • Ticket conversation. Threaded replies on a ticket — a public reply (visible to the customer) and an internal note (agents only).
  • List + filters + search. A ticket list/board filterable by status, priority and assignee, with text search on subject/body.
  • REST API + React frontend. A documented JSON API consumed by your React app (no server-rendered-only shortcuts for the core flows).
  • Seeded demo data. A seeder that creates 1 org, an admin, 2 agents, 2 customers and ~12 tickets so a judge sees a populated app instantly.

Should Depth — where good builds pull ahead

  • SLA policies & timers. Per-priority response & resolution targets; a live “time remaining / breached” indicator on each ticket.
  • Queues & assignment. “Unassigned”, “My tickets”, reassignment; an agent can claim a ticket.
  • Activity log / audit trail per ticket (status change, assignment, replies) with who + when.
  • Dashboard metrics. Open by status/priority, avg first-response time, SLA-breach rate, tickets created per day.
  • Notifications. An in-app notification centre (or a logged/webhook notification) when a ticket is assigned or replied to.

Stretch Wow — only if your sprints are flying

  • Canned responses / macros · ticket merge · CSAT rating after resolution.
  • A separate customer portal view · bulk actions · a public, rate-limited “submit a ticket” API.
  • Real-time updates (polling or websockets) · full-text search · CSV export.
This is the brief — build PulseDesk, not something else. Building a different app is the single most common way to get disqualified. If a feature is ambiguous, make a sensible call, write it in your README, and move on.

05 · The agile AI-agent workflow

This is the heart of the event. Get this right.

Your agents — 2 required, more earn bonus

The core is the Hermes + OpenClaw pair. You are not limited to two — adding a reviewer or a CI/deploy agent is how real AI-native teams actually run, and it earns bonus marks (§08). But quality beats quantity: a tight 2-agent loop scores higher than a sloppy 4-agent one.

Your Hermes config

ChannelPurpose

#agent-coderHermes ↔ OpenClaw. Task handoffs & clarifications.

The sprint loop you must run (≥ 2 sprints)

  1. Spec → backlog. Tell Hermes the goal; have it produce a sprint backlog of small issues before any code is written. Save it to sprints/sprint-0N.md.
  2. Assign. Hermes hands a scoped issue to OpenClaw in #agent-coder.
  3. Build + report. OpenClaw implements, runs tests, opens a PR, and posts its report to #agent-log.
  4. CI. GitHub Actions runs on the PR; results land in #ci-cd.
  5. Human review & merge. You review the PR in #human-review and merge it yourself. No bot merges to main.
  6. Repeat with the next tight sprint. Aim for 3–4 small sprints rather than 1 big one.
Minimum bar: at least 2 real sprints, each with a saved sprint doc, a genuine Hermes→OpenClaw handoff in Slack, a PR you merged, and OpenClaw reports in #agent-log. More tight sprints = a stronger process score.
Human-in-the-loop is mandatory. Agents must communicate through Slack channels, not silently. You merge to main. If you let a bot run unsupervised straight to main, you lose the entire process score.

Hard gates & caps

  • DQ: no public/clonable repo · built the wrong thing (not PulseDesk) · fabricated evidence (your Slack / git / CI timelines contradict each other) · >75% plagiarised or a forked existing app passed off as yours.
  • Capped low if: no genuine two-agent loop (it's just hand-coded) · a confirmed cross-tenant data leak · the app is dead / localhost-only / frontend-with-no-backend.

  • Not the required stack (no Laravel + MySQL backend / no React app UI).

How we verify (so you know it's fair)

Judges clone your repo and run migrate --seed + your frontend; read your configs, agent-log.md and sprint docs; open your Slack evidence; and cross-check three independent timelines — your Slack export, your git/PR history, and your CI runs — which must agree. We use server-side GitHub timestamps (commit dates are forgeable, so they don't count) and confirm you are the PR merge actor. We run an adversarial multi-tenant probe (sign in as Org A, try to read Org B). Because it's in-person, a judge may ask you to run it live from a fresh clone. Borderline scores get a second independent review, and the top builds are hand-validated before any winner is announced. You'll get a transparent written breakdown on the results page, exactly like the qualifier.

The honest path always wins. A modest app with a real, well-documented agent loop scores far better than an impressive-looking app with no visible orchestration or with faked evidence. Build real, log everything.

10 · FAQ

Every doubt, pre-answered.

Do I have to finish every feature?

No. Land the Must tier solidly, then go as deep as your sprints allow. We score how far you got and how cleanly you ran the agent process. A finished Must-tier with a great loop beats a half-broken Stretch attempt.

Do I have to deploy it live?

A live URL is a bonus, not a hard gate. What's required is that your app actually runs — either we can run it from a fresh clone using your README steps (composer install, migrate --seed, npm run dev), or you give a working live URL. If you deploy, point your frontend at the deployed API (not localhost).

Where do my screenshots, Slack export, and video go?

All inside your Git repo — nothing external. Configs in agents/, the loop in agent-log.md, sprint docs in sprints/, Slack proof in slack-export/ (export) or slack-export/screenshots/, and app/agent/CI screenshots in evidence/screenshots/ (see §06–§07 for exact paths). There is no video this year. We do not accept Google Drive, Loom, YouTube or any external link — if it isn't committed to the repo, it isn't counted. The submission form only asks for your repo URL.

Why no video / no Drive this time?

It keeps judging fair, fast and permanent (links rot and sign-in walls cost people points last time), and it's an in-person event — a judge will just ask you to run it live, which beats any recording.

How exactly do I get my $50 EastRouter credit?

Sign up at dashboard.eastrouter.com with Google or GitHub, then share the email/handle you signed up with to your Forge captain. They top you up to $50. A $10 trial works instantly so you're never blocked while that happens. (See §02.)

Which model should I use?

z-ai/glm-5.1 is the recommended default for both planning and coding. For heavy architecture/planning, give Hermes deepseek/deepseek-v4-pro (1M context, strong reasoning). For cheap routine edits, drop to z-ai/glm-4.5-air or deepseek/deepseek-v4-flash. Full list in §02.

Can I use Claude Code / Codex / Cursor / Cline as the coder instead of OpenClaw?

The event is specifically about the Hermes (orchestrator) + OpenClaw (coder) pairing over Slack — that's what we grade and what your $50 is for. Use other tools only as a personal editor; the graded loop must be Hermes → OpenClaw. (EastRouter does support Claude Code / Codex if you want them as helpers.)

What counts as a real “agent loop”?

A genuine exchange in Slack where you give Hermes a goal, Hermes plans & assigns a scoped task to OpenClaw, OpenClaw implements + reports back, and you review/merge. Real messages, real PRs, real commits — not a single screenshot of “bot joined the channel,” and not a hand-written log pasted in after the fact.

An agent broke / hit a rate limit / produced garbage. What do I do?

That's normal — handling it is the skill. Re-scope the task smaller, switch to a different EastRouter model, fix the config, and note what happened in agent-log.md. Honest recovery from failure scores better than pretending it went perfectly.

Can I use a UI kit / component library / Laravel starter kit?

Yes — scaffolds, Tailwind UI, shadcn-style kits, Breeze/Jetstream for auth, etc. are all fine. The PulseDesk features must be built by your agents during the event; boilerplate doesn't have to be.

Do I need MySQL specifically? I used SQLite in the qualifier.

Yes, MySQL 8 for the final build (it's the spec, and it's a more realistic multi-tenant DB). Use migrations + seeders so it sets up cleanly.

Should my repo be public or private?

Public. A private/missing repo can't be cloned and is a hard gate. Make it public before you submit and keep it public until results are out.

What if I'm not done at 6:00 PM?

The form doesn't auto-close, so you can still submit — but the clock is real and late submissions are reviewed at the organizers' discretion. Better to land a clean, smaller build and submit on time than to chase a big one past the buzzer.

How and when do I find out if I won / how I did?

Same as the qualifier: a results page where you enter your GitHub username and see your status + full, transparent score breakdown and feedback. Watch the Forge Slack / your email.

Can I resubmit?

Yes — submit again and we score your latest entry. Use the same GitHub username each time so we can match it.

Something here is ambiguous / I have a doubt not covered.

Make a reasonable decision, write it in your README, and keep moving — that judgement is part of the test. For anything blocking, ask a Forge captain in person or in the Forge Slack.
